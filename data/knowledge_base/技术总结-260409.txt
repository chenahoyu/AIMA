项目名称：基于人工智能的 grandMA3 离线语义编程助手技术方案

1. 核心需求分析
本方案旨在为 grandMA3 离线编程环境构建一套语义化辅助系统，将传统的基于通道和物理地址的编程模式转化为基于自然语言意图的逻辑控制。优化版本在原需求基础上，进一步强化了系统的容错能力、AI 输出确定性及工程落地可行性。
1.1 语义化操作
实现通过自然语言指令（如：让全场灯光呈现忧郁的蓝色调）自动检索并执行对应的控制路径，降低对特定 DMX 数值记忆的依赖。优化版采用结构化输出机制，确保 AI 翻译结果与 OSC 控制原语严格对应。
1.2 实时状态感知
系统需具备读取 grandMA3 当前运行状态的能力，包括灯具选择、序列激活状态及预设数值，确保 AI 指令基于当前的编程上下文而非盲目下发。优化版引入双通道状态获取策略：主用 Socket 增量推送，备用文件系统轮询，提升连接鲁棒性。
1.3 离线工作流优化
针对离线编程场景，侧重于批量指令生成、自动 Patch 校验以及创意提案的快速落地，而非追求演出现场的毫秒级同步。新增“干运行”预览模式，确保所有自动化指令在人工确认后下发。
1.4 指令安全性与准确性
建立校验机制，确保 AI 生成的 OSC 指令符合 grandMA3 语法规范，并能准确映射至现有的灯具属性及宏命令。优化版采用 Function Calling 结构化约束与 Python 端二次范围校验的双层保险架构。

2. 技术实现路径
2.1 数据同步逻辑：双通道镜像与降级容错
系统采用主备结合的状态同步策略，确保内存镜像在离线环境下的持续可用性。
主通道（Socket 增量推送）：通过 Lua 插件调用 grandMA3 API 监控 SelectedSequence、ActiveExecutors 等活跃对象，利用 TCP Socket 将状态变动以轻量化数据包推送至 Python 后端。为降低控制台性能开销，推送频率限制为 200ms 合并周期。
备用通道（文件系统轮询降级）：当 Socket 连接中断或 Lua 网络扩展不可用时，Lua 侧自动切换至周期性 ExportJson 导出模式，将 Patch、Sequence 基础数据写入共享目录。Python 端利用 watchdog 监听文件变更，实现降级状态更新。此设计确保系统在纯离线单机作业中的健壮性。
镜像模型精简：Python 内存镜像仅维护语义决策必需的最小数据集，包括 FixtureID→属性映射表、Sequence→Cue 结构及 Group 成员列表，降低 Diff 计算开销与同步复杂度。

2.2 决策大脑：LLM 语义翻译层（结构化增强）
上下文构建：将精简镜像中的状态信息转化为提示词上下文，注入当前选中灯具、激活序列及临近 Cue 的色调倾向。
结构化输出约束：摒弃纯文本 OSC 路径生成方式，全面采用 OpenAI Function Calling 或 JSON Mode 强制 AI 返回结构化意图描述。示例如下：
```json
{
  "intent": "set_fixture_color",
  "fixture_ids": [101, 102, 103],
  "color_description": "deep_blue",
  "suggested_rgb": [0, 20, 80],
  "additional_actions": ["ensure_dimmer_on"]
}
```
Python 层根据结构化数据组装合规 OSC 指令，彻底消除语法错误与幻觉输出。
动作模板库：建立常用操作（选灯、调色、存 Cue、执行宏）的预定义动作模板，AI 仅需填充模板参数，进一步约束输出空间。

2.3 校验与执行层：干运行预览与 Schema 验证
路径校验与范围检查：在 OSC 指令组装阶段，Python 后端对照内存镜像验证目标对象（灯具、序列、Cue）的存在性，并对 Dimmer、RGB 等属性值进行 0-100 或 0-255 范围裁剪。
干运行预览机制：所有由 AI 生成的指令序列首先进入 Streamlit 交互界面展示，内容包括等效 Lua 命令行、即将发送的 OSC 地址与参数。用户确认后，系统通过 python-osc 库执行最终触发。该机制提供最终人工决策节点，保障离线编程的安全底线。
连接预检：下发指令前发送轻量级 `/gma3/cmd "Lua 'Printf(\"healthcheck\")'"` 验证控制台 Lua 执行环境就绪，异常时自动切换至文件降级模式并提示用户。

2.4 个性化知识沉淀：轻量化本地偏好学习
记录用户对 AI 指令的手动修正日志（如：忧郁蓝意图下，AI 建议 RGB(0,0,100) 被用户修正为 RGB(20,10,80)）。
采用本地向量数据库（ChromaDB）存储修正前后的意图-数值对。后续处理相似语义意图时，检索最近邻修正记录作为 Few-shot 示例注入 Prompt，使 AI 输出逐渐趋近用户个人设计习惯。整个过程完全离线运行，无需依赖云端 Fine-tuning。

3. 技术栈构成
3.1 控制台端 (grandMA3 Side)
核心引擎：grandMA3 v2.0+ 软件环境。
脚本语言：Lua（调用内部 API、执行 ExportJson 导出、维护 Socket 客户端）。
通信协议：OSC over UDP (默认端口 8000)；TCP Socket (主状态推送)；共享文件夹（备用状态交换）。
3.2 中间件层 (Python Side)
核心开发语言：Python 3.10+。
通信库：python-osc (指令下发)；socket / asyncio (状态接收)。
逻辑处理：watchdog (备用文件监听)；Pydantic (结构化数据校验)；Streamlit (预览确认界面)。
内存镜像：基于 Dataclass 构建的轻量化状态容器。
3.3 智能层 (AI Side)
大语言模型：GPT-4o 或同级别模型，需支持 Function Calling 或 JSON Mode。
接口规范：OpenAI API 兼容接口。
本地知识库：ChromaDB (轻量化向量检索，用于偏好学习)。
开发环境：macOS (MacBook / Mac mini)，与 grandMA3 onPC 运行于同一主机或局域网内。

4. 预期目标与交付效果（优化版）
4.1 生产力提升
通过自然语言快速生成基础 Cue 列表和颜色布局，减少手动输入命令行的频率。结合结构化预览机制，预计将基础框架搭建时间缩短 50% 以上，同时显著降低因 OSC 语法错误导致的调试成本。
4.2 逻辑闭环与鲁棒性增强
系统不仅理解“因为 A 灯具关闭，调色需附带开光”的上下文逻辑，还能在网络波动或 Lua 环境受限时无缝降级至文件交换模式，确保辅助功能持续在线。
4.3 确定性执行与安全可控
借助 Function Calling 与干运行预览，AI 生成的每一条指令均经过路径校验与用户目视确认，将误操作风险降至行业可接受的最低水平。
4.4 个性化知识沉淀
通过本地向量检索实现无感偏好学习，长期使用后系统输出的色调方案与操作序列将更贴合设计师个人风格，形成差异化的智能辅助体验。

5. 总结
本优化方案在保留原架构语义交互核心理念的基础上，针对工程落地中的状态同步脆弱性、AI 输出不确定性与执行安全性三大痛点进行了针对性加固。通过双通道状态镜像、结构化输出约束、干运行预览及本地偏好学习四项关键优化，系统在离线编程场景下的鲁棒性、准确性与个性化程度均获得显著提升，为 grandMA3 灯光设计师提供了一个可靠、安全且能持续进化的语义化编程助理。
